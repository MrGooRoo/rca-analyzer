## 1. Резюме

| Категория | Оценка | Комментарий |
|-----------|--------|-------------|
| Архитектура | 7/10 | Чистая слоистая структура, Repository pattern, DI; есть утечки абстракций (commit в read) |
| Backend | 6/10 | Работоспособно, но есть критические баги (SSE, import, shared httpx) |

**Вердикт:** продуманный каркас приложения с хорошим фундаментом безопасности, но с накопленным техдолгом в frontend-слое и серией критических runtime-багов, часть из которых молчаливо ломает функционал (SSE-дедупликация, admin-эндпоинт). Безопасность в production защищена fail-fast-проверками, но dev-окружение скомпрометировано дефолтным секретом.

## 2. Сильные стороны

### 2.1. Слоистая архитектура
- API → Services → Domain → DB с чистым разделением ответственности.
- Domain не зависит от FastAPI; методики — плагины через `MethodologyRunner` (`src/domain/methodologies/`).
- Промпты вынесены в `configs/prompts/*.j2` (Jinja2).
- Интеграции изолированы в `src/integrations/` (LLM, embeddings).

### 2.2. Repository pattern и DI
Единая точка работы с БД через репозитории + `Depends()` (FastAPI). Автоматические Alembic-миграции (20 версий).

### 2.3. Auth и CSRF
- JWT (access) + refresh token rotation с револьвером и `revoked_at`.
- httpOnly cookies, signed double-submit CSRF (HMAC, constant-time compare).
- Account lockout после `MAX_FAILED_LOGIN_ATTEMPTS` (5) неудач на `LOCKOUT_MINUTES` (15).
- bcrypt напрямую (без passlib).

### 2.4. Multi-methodology анализ
5 методик (Five Why, Ishikawa, FTA, Bowtie, RCA Systemic) с SSE-streaming прогресса.

### 2.5. Embeddings и поиск похожих инцидентов
pgvector + три провайдера эмбеддингов (local hash, HuggingFace, OpenRouter) с фабрикой и автофолбэком.

### 2.6. Frontend: отмена анализа
`AbortController` + `analysisRunRef`, `beforeunload`, confirm при уходе со страницы во время анализа.

## 4. Высокие проблемы (P1)

### 4.1. Generic `Exception` handler перехватывает `HTTPException`
`src/api/app.py:148` — клиент получает 500 вместо корректного 403/404. Нужно либо зарегистрировать обработчик `HTTPException` первым, либо проверять `isinstance`.

### 4.2. Результаты без `user_id` доступны любому
`src/api/routes/export.py:39` — короткое замыкание `result.user_id and ...` пропускает проверку, если `user_id is None`.

### 4.3. `AttributeError` при удалённом пользователе
`src/api/routes/models.py:171,195` — `user.model_preferences` без проверки `user is None`.

### 4.4. Shared httpx-клиент фиксирует API-ключ первого экземпляра
`src/integrations/llm/openrouter.py:141` — критично для multi-provider. Второй клиент с другим ключом использует чужой.

### 4.5. CSRF exempt на `/api/v1/auth/refresh`
`src/api/middleware/csrf.py:53` — комбинируется с `allow_credentials=True`, создаёт вектор атаки через malicious-страницу.

### 4.6. In-memory rate limiter неэффективен
`src/api/middleware/rate_limiter.py:29` — при >1 worker каждый имеет свой `_store`; лимит обходится. Не покрывает ресурсоёмкие endpoints (`/analyze`, `/upload-report`).

### 4.7. API-ключи провайдеров хранятся в БД в plaintext
`src/db/orm_models.py:347` — при компрометации БД все ключи раскрыты. Нужны AES-GCM или vault.

### 4.8. Cookie `Secure` по умолчанию выключен
`src/auth/cookies.py:18` — `AUTH_COOKIE_SECURE=false`. В HTTPS-окружении без явного включения cookies уходят по HTTP.

### 4.9. Race condition в wallet
`src/services/wallet_service.py:77-107` — между чтением баланса и списанием возможна конкурентная модификация.

### 4.10. HF-embedding: ошибка загрузки кэшируется навсегда
`src/integrations/embeddings/hf_local.py:114` — временный сетевой сбой делает все последующие запросы падающими до перезапуска.

## 6. Frontend & UX

### 6.1. Архитектура
- **God Component**: `App.jsx` — 17 `useState` + 2 `useRef`; любое изменение ререндерит всё. Рекомендация: `useReducer` или Zustand/Jotai.
- **Нет роутинга**: при обновлении страницы теряется состояние (история/результат/админка).
- **Нет code splitting**: 6 страниц eager-import — нужен `React.lazy()`.
- **Глобальный mutable state** в `api.js` вне React-экосистемы.
- **Stale closures** в `useEffect`: `AnalysisProgress.jsx`, `ModelSelector.jsx`, `VictimSection.jsx`.
- **Мёртвый код**: `SimilarIncidentsHint.jsx`, `CausalTree` в `ResultView.jsx`, `FtaNode` в `fta.py`.

### 6.2. Доступность (a11y) — 8 проблем
| # | Проблема | Где |
|---|----------|-----|
| 1 | Модальные окна без `role="dialog"`, `aria-modal`, focus trap | `AdminPage.jsx`, `ConfirmDialog.jsx` |
| 2 | Кликабельные карточки без `role="button"`, `tabIndex`, `onKeyDown` | `HistoryPage.jsx:177,214` |
| 3 | Theme toggle без `aria-label`/`aria-pressed` | `App.jsx:379` |
| 4 | Toast без `aria-live` | `ui/Toast.jsx:42` |
| 5 | Прогресс-бары без `role="progressbar"`, `aria-valuenow` | `AnalysisProgress.jsx:108`, `SingleAnalysisProgress.jsx:50` |
| 6 | Контраст ошибок ~2.5:1 (WCAG AA требует 4.5:1) | `AuthPage.css:85`, `ResultView.css:58` |
| 7 | ErrorBoundary хардкод белого фона — не для тёмной темы | `ErrorBoundary.jsx:20` |
| 8 | `outline: none` на `:focus-visible` убирает стандартный фокус | `index.css:73` |

### 6.3. Адаптивность и CSS
- Header переполняется на < 480px; SVG-диаграммы (Ishikawa 900×400, FiveWhy 800) не адаптируются.
- Глобальный `transition` на `*, *::before, *::after` — риск задержек в списках.
- Hardcoded цвета вместо CSS-переменных; дублирование `@keyframes fadeIn`; неиспользуемые классы.

### 6.4. Тесты
- Frontend: 4 smoke-теста. **0% бизнес-логики** (auth, формы, SSE, экспорт, админка не покрыты).
- Backend: wallet service и cache_repository не протестированы.

## 8. Заключение

Проект обладает крепким фундаментом (архитектура, методики, embeddings, базовый security-слой) и готов к дальнейшему развитию. Основные риски на сегодня:

1. **Runtime-баги** P0 (#1, #2, #6) молчаливо ломают пользовательский функционал — высокая видимость для пользователей.
2. **Безопасность dev-окружения** полностью скомпрометирована дефолтным `JWT_SECRET`; production защищён, но защита держится на одном флаге `RCA_ENV`.
3. **Frontend** накопил техдолг: God Component, отсутствие роутинга и критически низкое тестовое покрытие.
4. **a11y** — 8 проблем, совместимых с WCAG 2.1 AA, при этом проект позиционируется как внутренний инструмент (вероятна правовая необходимость соответствия).

Закрытие P0-списка (6 пунктов) восстанавливает корректность ключевых потоков и поднимает планку безопасности. Дальнейшая работа идёт по P1/P2 согласно приоритетам.
