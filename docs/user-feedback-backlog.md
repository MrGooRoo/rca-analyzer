# User feedback backlog — 2026-06-14

> Первая партия пользовательских замечаний по текущему UI/UX RCA Analyzer. Приоритеты: P0 — критично, P1 — важно, P2 — улучшение, P3 — исследование/polishing.

## P1 — ближайшая доработка формы анализа

1. **Источник данных: вручную или DOCX + дозаполнение**
   - Нужна понятная логика: пользователь может заполнить вручную или загрузить DOCX.
   - Если DOCX содержит не все поля, найденные поля подставляются, отсутствующие пользователь дозаполняет вручную.
   - UI должен прямо объяснять это поведение.

2. **Placeholder-подсказки в ключевых полях**
   - Добавить нейтральные серые подсказки в «Заголовок инцидента», «Описание» и последующие поля.

3. **Убрать реальные примеры из placeholder**
   - Заменить «Цех №3, участок сборки», «ООО “ПромБезопасность”», «Производственный цех» на нейтральные подсказки.

4. **Поэтапный ввод данных**
   - Этапы: 1) Ввод исходных данных, 2) Выбор методологии анализа, 3) Результаты анализа.
   - Нужен индикатор этапов. Полноценный wizard можно делать отдельным патчем.

5. **Единый подход к заголовкам формы**
   - Привести разделы и подразделы к единому стилю.
   - Исправить нумерацию в блоке установленных фактов: сейчас есть 3.1, 3.2, 3.4, 3.5, 3.6 и нет 3.3.

6. **Параметры анализа как переключатель**
   - Режимы «Одна методика» / «Сравнить методики» должны выглядеть как логичный переключатель.

9. **Убрать рывок страницы при выборе второй методики**
   - Подсказка «выберите минимум 2» не должна исчезать и менять высоту блока.
   - Лучше сделать постоянную строку статуса выбора.

10. **Единый формат названий методик**
    - Привести названия к одному формату во всех местах UI.

11. **Порядок и default методики**
    - Порядок: 5 Почему → BowTie → Ishikawa → FTA → RCA системный.
    - По умолчанию: 5 Почему.

12. **Сохранение введённых данных при переходе Анализ ↔ История**
    - При возврате на вкладку «Анализ» черновик формы должен сохраняться.

15. **Статус выполнения для одиночного анализа**
    - Сейчас статус есть для multi-analysis, но не для одной методики.

## P1/P2 — результаты и история

7. **Динамическая ширина «Различающихся выводов»**
   - Если методик 2–3, карточки должны занимать всю ширину без пустоты справа.

8. **Переосмыслить индикатор похожих инцидентов**
   - Убрать фразу «подробности покажем после анализа».
   - Найти логичный способ показать наличие похожих случаев без перегрузки формы.
   - Возможный вариант: компактная ссылка/индикатор в форме и полноценный блок под результатом текущего анализа.

13. **Кнопка отмены анализа**
   - Текущий отдельный блок визуально тяжёлый и прилегает к форме.
   - Нужно встроить отмену эстетичнее: в панель статуса/прогресса или sticky footer формы.

14. **Шкала/процент обработки анализа**
   - Сейчас в основном состояния «выполняется/готово».
   - Для multi-analysis есть прогресс по методикам, для single нужен хотя бы этапный progress indicator.

18. **Упростить оформление страницы результата**
   - Убрать текст «Форма скрыта...».
   - Сократить уровни вложенности карточек.
   - Развести текущий результат, похожие инциденты и метаданные более чисто.

19. **Кнопка «Открыть» в похожих инцидентах**
   - Разъяснить назначение: открывает сохранённый RCAResult похожего инцидента из истории.
   - Нужно решить: оставить, переименовать или заменить на более понятное действие.

20. **BowTie visualisation**
   - Текущие карточки BowTie выглядят хаотично.
   - Нужно вернуться к визуализации «галстук-бабочка» или предложить альтернативную структурную схему.
