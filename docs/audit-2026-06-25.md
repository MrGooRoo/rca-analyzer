## 📋 Резюме

RCA Analyzer — это веб-приложение для анализа корневых причин (Root Cause Analysis) промышленных инцидентов. Проект поддерживает 5 RCA-методик (5 Why, Ishikawa, FTA, Systemic RCA, Bowtie), использует LLM через OpenRouter для генерации анализа, pgvector для поиска похожих инцидентов и DOCX/PDF экспорт.

**Общая оценка:** 🔶 **7.2 / 10** — проект функционально богат и технически зрел для своего возраста (~3 недели), но до продакшена требуется серьёзная работа по безопасности, наблюдаемости и UX-полировке.

## 1. 🏗️ АРХИТЕКТУРА — 7.5/10

### ✅ Сильные стороны

| **Доменные модели** | ⭐⭐⭐⭐ | Pydantic v2 модели в `src/domain/` отделены от ORM-моделей в `src/db/`. Нет утечки БД-логики в бизнес-слой. |

| **Миграции** | ⭐⭐⭐⭐⭐ | 20 миграций Alembic, включая data-migrations (backfill). Это говорит о зрелом подходе к эволюции схемы. |

### ⚠️ Проблемы и рекомендации

#### P0 — Критические

**1. God Component: `App.jsx` (~350+ строк состояния)**
```
page, result, comparison, viewMode, loading, multiProgressPayload,
analysisSignal, formDraft, singleProgress, analysisRunRef,
abortControllerRef, walletBalance, isAdmin
```
Это 12+ стейт-переменных в одном корневом компоненте. Это анти-паттерн, который:
- Делает компонент нечитаемым
- Приводит к лишним рендерам
- Затрудняет тестирование

**Решение (Apple-style):** Разделить на конечный автомат (state machine) через `useReducer` или библиотеку типа `xstate`:
```
IDLE → FILLING_FORM → ANALYZING → SHOWING_RESULT → VIEWING_HISTORY
```

**2. Отсутствие State Management слоя**
Проект вырос за пределы "одного компонента", но всё ещё использует prop drilling и `useState` в App. Нет ни Context API для глобального состояния (кроме AuthContext), ни Zustand/Jotai.

**Решение:** Минимум — 3 контекста:
- `AuthContext` (уже есть ✅)
- `AnalysisContext` (состояние анализа + результаты)
- `WalletContext` (баланс, транзакции)

**3. Сервисный слой без интерфейсов/протоколов**
`src/services/` содержит конкретные классы без абстракций. Это затрудняет:
- Мокирование в тестах (приходится патчить)
- Замену реализаций (например, другой LLM-провайдер)

**Решение:** Ввести `Protocol`-ы (Python 3.11):
```python
class LLMProvider(Protocol):
    async def complete(self, prompt: str, ...) -> str: ...
```

#### P1 — Важные

**4. Нет слоя кэширования на уровне сервиса**
DOCX-кэш реализован через отдельную таблицу `docx_extraction_cache` — это хорошо. Но нет HTTP-кэширования (ETag, Cache-Control) для результатов анализа, которые не меняются.

**5. `api.js` — монолитный файл на ~400+ строк**
Централизованный API-клиент — это ок, но он уже перерос один файл. Стоит разделить на модули:
```
api/
  client.js      — базовый fetch + refresh + CSRF
  auth.js        — login, register, refresh, logout
  analysis.js    — analyze, analyzeStream, multi
  wallet.js      — get, topUp
  export.js      — exportDocx, exportPdf
```

**6. Отсутствие Graceful Degradation на фронтенде**
Если OpenRouter недоступен — пользователь видит toast с ошибкой и всё. Нет offline-поддержки, нет queue для повторных попыток.

## 2. 🔒 БЕЗОПАСНОСТЬ — 6.0/10

### ✅ Что сделано хорошо

| Механизм | Статус | Качество |
|----------|--------|----------|
| **JWT access + refresh rotation** | ✅ | httpOnly cookies, SHA-256 хэш refresh в БД, ротация при каждом использовании |
| **CSRF (signed double-submit)** | ✅ | HMAC-SHA256 подпись, constant-time сравнение, двухфазная защита login/register |
| **Account lockout** | ✅ | 5 попыток → 15 мин блок, авто-сброс |
| **Rate limiting** | ✅ | Sliding window, 10 req/15 min по IP |
| **bcrypt хэширование** | ✅ | Напрямую (без passlib), `gensalt()` — корректно |
| **Ролевая модель** | ✅ | admin/user, self-demotion защита |
| **Pydantic max_length** | ✅ | Защита от OOM на строковых полях |
| **SQL Injection** | ✅ | SQLAlchemy ORM + parameterized queries |

### 🔴 Критические уязвимости

**1. Hardcoded credentials в docker-compose.yml**
```yaml
POSTGRES_USER: rca
POSTGRES_PASSWORD: secret   # ← В ОТКРЫТОМ ВИДЕ В РЕПОЗИТОРИИ
```
Пароль БД захардкожен в файле, который коммичен в Git. Любой, кто клонирует репо, знает пароль.

**Решение:** Вынести в `.env` (уже есть `DATABASE_URL` в env, но дублируется в compose):
```yaml
environment:
  - DATABASE_URL=${DATABASE_URL}
```

**2. In-memory Rate Limiter — не переживает рестарт**
```python
# src/api/middleware/rate_limiter.py — sliding window в dict
```
При перезапуске API все лимиты сбрасываются. Атакующий может просто подождать рестарта.

**Решение:** Redis-backed rate limiter (даже для MVP можно in-DB через `rate_limit_entries` таблицу).

**3. Отсутствие Security Headers**
Нет ни одного из стандартных заголовков безопасности:
```
X-Content-Type-Options: nosniff          — нет
X-Frame-Options: DENY                    — нет  
Strict-Transport-Security                — нет
Content-Security-Policy                  — нет
Referrer-Policy                          — нет
Permissions-Policy                       — нет
```

**Решение:** Добавить middleware:
```python
@app.middleware("http")
async def security_headers(request, call_next):
    response = await call_next(request)
    response.headers["X-Content-Type-Options"] = "nosniff"
    response.headers["X-Frame-Options"] = "DENY"
    response.headers["Referrer-Policy"] = "strict-origin-when-cross-origin"
    return response
```

**4. JWT_SECRET с fallback на слабое значение**
```python
SECRET_KEY: str = os.environ.get("JWT_SECRET", "change-me-in-production-please")
```
Если переменная не задана — приложение МОЛЧАЛИВО запускается с известным секретом. Нет валидации при старте.

**Решение:** Crashing при отсутствии в production:
```python
if os.environ.get("ENV", "dev") == "prod" and SECRET_KEY == "change-me-in-production-please":
    raise RuntimeError("JWT_SECRET must be changed in production!")
```

### 🟡 Средние риски

**5. Нет защиты от timing attack при проверке email**
```python
existing = (await db.execute(select(UserORM).where(...))).scalar_one_or_none()
if existing:
    raise HTTPException(status_code=400, detail="Email already registered")
```
Разное время ответа для существующего и несуществующего email позволяет enumeration.

**Решение:** Одинаковое время ответа (или общий текст ошибки: "Если email зарегистрирован, письмо отправлено").

**6. CORS с wildcard-риском**
```env
CORS_ALLOW_ORIGINS=http://localhost:5173,http://localhost:3000
```
В проде нужно строго ограничить. Нет валидации что Origins не содержат `*`.

**7. Нет Password Complexity Policy**
Минимальная длина 10 символов — это хорошо, но нет требования: цифра, спецсимвол, верхний/нижний регистр.

**8. Refresh Token без family tracking**
Если refresh token украден, легитимный пользователь не узнает. Нет detection-механизма: если использован revoked token → вся семья токенов инвалидируется.

Это **огромный** минус для профессионального приложения. Многие корпоративные пользователи работают в светлом режиме.

**3. Нет Onboarding / First Run Experience**
Новый пользователь видит пустую форму без подсказок. Apple всегда проводит через первый опыт использования.

**Решение:**

- Pre-filled пример инцидента (как в Apple Notes — первый заметка "Welcome")
- Skeleton loaders при загрузке

**4. Форма инцидента — перегружена**
Множество полей без визуальной группировки. Apple разбивает длинные формы на шаги (Wizard / Stepper).

**Решение:** Multi-step wizard:
```
Шаг 1: Что произошло? (title, description, date, location)

Шаг 3: Пострадавшие и действия (victims, actions_taken)

```

**5. Нет Keyboard Shortcuts**
Для профессионального инструмента обязательно:

- `Esc` — отмена

**6. Результат анализа — "стена текста"**
RCA-результат выводится как длинный текст. Apple использует:
- Collapsible sections
- Visual diagrams (Ishikawa — рендерить как диаграмму, не текст!)
- Summary cards с ключевыми метриками
- Color-coded severity indicators

### 🟡 Средние UX-проблемы

**7. Нет Confirmation Dialog перед удалением**
В админке кнопка удаления без подтверждения — нарушение Apple HIG.

**8. Нет Undo для критичных действий**
"Новый анализ" сбрасывает форму без возможности вернуть.

**9. Wallet Balance — неинтуитивно**
Баланс в хедере без контекста. Что означает $0.01? Сколько стоит анализ? Нет pricing page.

**10. Нет Localization / i18n**
Вся строка на русском — это ограничивает аудиторию. Но для промышленного сектора РФ — пока приемлемо.

```jsx
// SVG-based Ishikawa
<IshikawaDiagram 
  effect="Падение с высоты"
  causes={{
    man: ["Недостаточный инструктаж", "Нарушение ТБ"],
    machine: ["Неисправность стремянки"],
```

```
          │
          └→ Почему? Нет в наличии
              │
              └→ Почему? Закупка не прошла
                  │
                  └→ Почему? Бюджет не утверждён ← ROOT CAUSE
```

## 7. 🗺️ ROADMAP: ОТ ХОРОШЕГО К APPLE-LEVEL
