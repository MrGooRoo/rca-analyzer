# methodologies.md — Описание методик RCA

> Документ описывает поддерживаемые RCA-методики, их назначение и особенности реализации в проекте.

## 1. Поддерживаемые методики

| Методика | Enum | Runner | Prompt | Статус |
|---|---|---|---|---|
| 5 Почему | `five_why` | `FiveWhyRunner` | `five_why.j2` | ✅ Готово |
| Диаграмма Исикавы | `ishikawa` | `IshikawaRunner` | `ishikawa.j2` | ✅ Готово |
| Fault Tree Analysis | `fta` | `FaultTreeRunner` | `fta.j2` | ✅ Готово |
| BowTie / Галстук-бабочка | `bowtie` | `BowTieRunner` | `bowtie.j2` | ✅ Готово |
| Системный RCA | `rca_systemic` | `RcaSystemicRunner` | `rca_systemic.j2` | ✅ Готово |

Все методики возвращают единый `RCAResult`, поэтому frontend может отображать их одинаково и сравнивать между собой.

## 2. Общий контракт runner

Каждая методика наследует `MethodologyRunner` из `src/domain/methodologies/base.py`.

```python
class MethodologyRunner(ABC):
    @property
    @abstractmethod
    def methodology_type(self) -> str: ...

    @abstractmethod
    async def run(self, request: AnalysisRequest, raw_llm_response: dict) -> RCAResult: ...

    def get_prompt_template_name(self) -> str:
        return f"{self.methodology_type}.j2"
```

Runner не вызывает LLM сам. Он получает уже валидированный словарь `raw_llm_response` и преобразует его в доменную модель `RCAResult`.

## 3. 5 Почему — `five_why`

**Назначение:** быстрый итеративный поиск первопричины через цепочку вопросов «почему?».

**Лучше подходит для:**

- относительно простых инцидентов;
- случаев с линейной причинной цепочкой;
- первичного анализа, когда нужно быстро получить гипотезу root cause.

**Особенности результата:**

- строит причинную цепочку;
- выделяет immediate, contributing и root causes;
- рекомендации обычно привязаны к найденной первопричине.

**Ограничение:** может упрощать сложные системные инциденты, где причин несколько и они связаны нелинейно.

## 4. Диаграмма Исикавы — `ishikawa`

**Назначение:** разложить причины по категориям «рыбьей кости».

**Лучше подходит для:**

- производственных и организационных инцидентов;
- случаев, где вероятны причины в разных областях: люди, оборудование, среда, процессы;
- поиска contributing causes.

**Типовые категории:**

- человек / персонал;
- оборудование;
- метод / процесс;
- материалы;
- среда;
- управление / организация.

**Особенности результата:**

- причины удобно группируются по категориям;
- помогает не зациклиться на одной версии причины.

## 5. Fault Tree Analysis — `fta`

**Назначение:** построить дерево отказов от верхнего события к базовым событиям.

**Лучше подходит для:**

- технических отказов;
- инцидентов с оборудованием;
- случаев, где важно понять комбинацию условий, приведших к событию.

**Особенности результата:**

- акцент на логической структуре отказов;
- root causes интерпретируются как базовые события;
- полезен для технических corrective actions.

**Ограничение:** хуже отражает социальные и организационные факторы, если они не описаны во входных данных.

## 6. BowTie / Галстук-бабочка — `bowtie`

**Назначение:** связать угрозы, top event, последствия и барьеры управления риском.

**Лучше подходит для:**

- HSE/industrial safety сценариев;
- анализа барьеров до и после события;
- подготовки профилактических и mitigative мер.

**Особенности результата:**

- дополнительно использует bowtie-структуру: threats, top event, consequences, barriers;
- валидирует целостность bowtie-модели предупреждениями, не ломая анализ;
- хорошо подходит для визуализации риска.

## 7. Системный RCA — `rca_systemic`

**Назначение:** выявить системные и организационные причины, а не только непосредственные ошибки.

**Лучше подходит для:**

- сложных инцидентов;
- повторяющихся проблем;
- анализа культуры безопасности, управления, обучения, процедур и контроля.

**Особенности результата:**

- больше внимания management/system factors;
- рекомендации часто направлены на процессные изменения;
- полезен как финальная методика после быстрых методов.

## 8. Multi-analysis и сравнение

Frontend поддерживает два режима:

- `single` — одна методика;
- `multi` — 2–5 методик по одному инциденту.

Для multi-analysis backend создаёт одну `analysis_session`, запускает методики параллельно и сохраняет все результаты с общим `session_id`.

Сравнение строится через:

```text
GET /api/v1/results/compare?session_id=...
```

Сравнение показывает:

- общие рекомендации;
- различающиеся причины по методикам;
- сводку;
- side-by-side детали каждого результата.

## 9. Когда выбирать какую методику

| Ситуация | Рекомендованные методики |
|---|---|
| Нужно быстро найти вероятную первопричину | `five_why` |
| Нужно разложить причины по областям | `ishikawa` |
| Технический отказ оборудования | `fta` |
| HSE-риск, барьеры, последствия | `bowtie` |
| Сложная повторяющаяся проблема | `rca_systemic` + сравнение с другими |
| Нет уверенности, какая методика лучше | multi-analysis 3–5 методик |

## 10. Как добавить новую методику

1. Добавить значение в `MethodologyType` в `src/domain/models.py`.
2. Создать `src/domain/methodologies/<new_method>.py`.
3. Унаследоваться от `MethodologyRunner`.
4. Реализовать `methodology_type` и `run()`.
5. Добавить prompt `configs/prompts/<new_method>.j2`.
6. Зарегистрировать runner в orchestration-слое анализа.
7. Добавить frontend metadata в `frontend/src/lib/methodologies.js`.
8. Обновить `docs/contracts.md`, этот файл и тесты.
